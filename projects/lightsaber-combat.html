Lightsaber Combat Framework

LuauCombatServer-Authoritative

Overview

A request/response combat system driving all lightsaber interactions — attacks, blocks, parries, and kill confirmation — between client input and server-validated outcomes, built to stay responsive under real player-count load.

Problem

Melee combat is latency-sensitive: players expect instant feedback on swings and blocks, but every hit has to be validated server-side or the game becomes trivially exploitable. The framework needed a clear enum of request types so client and server always agree on what a given signal means, without a giant branching mess on either side.

Architecture

Every combat action is a numbered request in CombatConstants.ENUM.REQUEST. The client fires a request, the server validates state (stamina, cooldowns, distance, target legitimacy) and responds with the resolved outcome, which the client then plays back as VFX/animation — the server never trusts the client's own read of "did this hit land." 

-- client: request a swing
LightsaberSignal:Fire(CombatConstants.ENUM.REQUEST.Swing, { targetId = target })

-- server: validate, then respond with resolved outcome
function Lightsaber:HandleSignal(player, request, arguments)
	if request == REQUEST.Swing then
		if not self:CanSwing(player) then return end
		local resolved = self:ResolveSwing(player, arguments)
		self.LightsaberSignal:Fire(player, REQUEST.SwingResult, resolved)
	end
end

Result

A single dispatch point per side keeps combat logic auditable — every new attack type is one enum entry and one handler branch, not a new remote. Server stays the single source of truth for hit resolution while client-side playback keeps combat feeling immediate.
